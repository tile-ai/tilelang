CORE_RISKS:
- The highest-risk assumption is that `LayoutInference` already understands the “early pipelined” IR shape. It does not appear to. `PipelinePlanning` removes `num_stages` and preserves only `tl_pipelined_num_stages` in [pipeline_planning.cc](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/src/transform/pipeline_planning.cc#L1613), and `InjectSoftwarePipeline` wraps rewritten loops in `AttrStmt("tl.pipeline_context_num_stages", ...)` in [inject_pipeline.cc](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/src/transform/inject_pipeline.cc#L830). But `LayoutInference` only tracks pipelined state from `For.annotations["num_stages"]` in [layout_inference.cc](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/src/transform/layout_inference.cc#L653), even though it defines the new key in [layout_inference.cc](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/src/transform/layout_inference.cc#L50). That makes the current plan incomplete and likely to fail on fragment/shared layout inference for pipelined kernels.
- The plan assumes the known `inject_pipeline.cc` fix is close to sufficient. It addresses one concrete failure mode around block read/write recomputation and `local.var` promotion in [inject_pipeline.cc](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/src/transform/inject_pipeline.cc#L1194), but it does not prove the earlier-pass refactor is semantically correct across all wrapper forms, aliasing cases, and pipeline metadata consumers.
- The warp-specialized tiled path still shows signs of partial migration. The file header says it requires `InstructionAnnotation` and `MultiVersionBuffer` in [producer_consumer_ws_tiled.cc](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/src/transform/producer_consumer_ws_tiled.cc#L16), while the implementation now does inline classification and internally invokes `MultiVersionBuffer()` in [producer_consumer_ws_tiled.cc](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/src/transform/producer_consumer_ws_tiled.cc#L832). That mismatch is a signal that hidden preconditions may still exist in edge paths.
- Passing CI alone may not catch the most important regression class here: kernels that still compile and run, but lose async overlap or warp-specialized correctness/performance because pipeline context is no longer visible to the right passes.

MISSING_REQUIREMENTS:
- The draft says to revert the example debugging change, but it omits reverting the compiler-side debug prints currently left in [phase.py](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/tilelang/engine/phase.py#L191). Those `print(mod)` calls are debugging artifacts and should not be part of acceptance.
- The plan does not explicitly require compatibility for all pipeline metadata forms. With the current order, existing passes need to handle at least `num_stages`, `tl_pipelined_num_stages`, and `tl.pipeline_context_num_stages`, not just the legacy loop annotation.
- There is no explicit coverage requirement for distinct kernel classes. This refactor needs separate validation for TMA warp-specialized, cp.async-only, SIMT global-to-shared, and non-pipelined kernels. Treating “pipeline kernels” as one bucket is too coarse.
- The draft does not call out wrapper-heavy loop bodies as a requirement, even though both `PipelinePlanning` and `InjectSoftwarePipeline` have bespoke logic for `LetStmt` and `IfThenElse` wrappers. Those are common miscompile surfaces.
- The draft does not mention performance invariants. For a GPU compiler pipeline refactor, “correct but much slower” is still a failure unless you explicitly decide otherwise.

TECHNICAL_GAPS:
- The biggest gap is metadata propagation semantics. `LowerTileOp` already consumes `tl.pipeline_context_num_stages` in [lower_tile_op.cc](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/src/transform/lower_tile_op.cc#L1123), but `LayoutInference` does not. That asymmetry strongly suggests the refactor was only partially completed.
- The plan freezes pass order, but it does not enumerate which existing passes must be updated to behave correctly under the new order. Without that list, the work risks turning into test-chasing instead of a coherent fix set.
- The `ProducerConsumerWarpSpecializedTiled` / `RestoreWSThreadExtent` interaction is delicate and under-specified in the plan. The current comments in [phase.py](/weka-hg/prod/deepseek/permanent/wanglei/tilelang/tilelang/engine/phase.py#L203) imply that `LayoutInference` and `LowerTileOp` must see one thread-extent view, and downstream simplification must see another. That is an architecture constraint, not an implementation detail, and it should be called out explicitly.
- The plan is weak on unsupported-pattern behavior. If some shapes are not valid under early planning/injection, the passes need either a sound fallback or a precise failure mode. Silent partial transformation is the worst outcome here.

ALTERNATIVE_DIRECTIONS:
- The best alternative within your stated constraints is to formalize “pipeline context compatibility” as the work item, not “fix failing tests.” Concretely: update existing pass implementations so any pass that cares about pipelined state recognizes all current metadata carriers. Tradeoff: more cross-pass plumbing, but it preserves the intentional ordering.
- A second option is to make early pipeline passes more conservative and fail loudly on unsupported IR shapes instead of trying to rewrite them. Tradeoff: some workloads may temporarily lose feature coverage, but you avoid silent miscompiles and make the support boundary explicit.
- If you are willing to relax constraint 2 later, the cleanest architecture may still be to keep only the passes that are naturally tile-op-level before `LayoutInference`, and leave lowering-sensitive transforms later. Tradeoff: cleaner semantics, but it conflicts with the current branch direction.
